## 四、主干 1：Analysis / Evidence / Graph 主链

### 步骤

1. 打开 `/evidence`
2. 上传文件：`01_analysis_records_mvtec.jsonl`
3. 等待 run 生成
4. 检查页面是否出现：
   - run 列表
   - run detail
   - case 列表
   - visual evidence
   - ranked root causes
5. 点击跳转或切到 `/graphs`
6. 检查是否能看到：
   - candidate list
   - path graph
   - evidence / graph linked inspection

### 通过标准

- 上传成功
- `/evidence` 能看到 run / case / detail
- `/graphs` 能看到 candidate / path
- 页面没有退回 mock-only 才能工作的提示

---

## 五、主干 2：Feedback / Ledger 主链

### 步骤

1. 在 `/graphs` 中选中一个 candidate 或 path
2. 将 `02_feedback_note_example.txt` 中的文本复制到反馈备注里
3. 提交反馈
4. 切到 ledger 区域
5. 检查刚刚的反馈是否被读回

### 重点检查

- `root_cause_candidate` 是否能作为正式反馈目标提交
- ledger 是否会显示 candidate 类型记录
- `target_key` 是否稳定存在

### 通过标准

- candidate / path / edge 至少有一类反馈能成功提交
- ledger 能读回刚提交的记录
- backend mode 下不再提示“review ledger 列表接口不存在”

---

## 六、主干 3：Materials / Extract / Draft / Build-sources 主链

### 步骤

1. 打开 `/materials`
2. 上传文件：`03_material_pump_note.txt`
3. 选中该 material
4. 执行 **extract**
5. 检查 tab/详情里是否能看到：
   - detail
   - chunks
   - extraction runs
   - artifacts
6. 点击生成 **source draft preview**
7. 检查是否有 preview 结果
8. 检查 **Draft History** 是否可读
9. 勾选该 material，执行 **build-sources / 生成 build request 并构图**

### 注意

- 这一步重点是验证：
  - material library 主链
  - extract 主链
  - draft history 主链
  - build-sources 主链
- **不强求最终 construction build 的新版本效果**，因为你说构图部分后续还会合并新版本。

### 通过标准

- material list 正常返回
- extract 成功
- chunks / extractions / artifacts 有数据
- source draft preview 有结果
- draft history 可读
- build-sources 能返回 construction-ready request，并触发后续构图入口

---

## 七、建议的联调顺序

推荐严格按这个顺序走：

1. **先走主干 1**，确认分析主链没问题
2. **再走主干 2**，确认反馈和 ledger 没问题
3. **最后走主干 3**，确认 materials 主链没问题

这样做的原因是：

- 主干 2 依赖主干 1 先产出 run / case
- 主干 3 是独立主链，但页面和 contract 更复杂，适合最后测

---

## 八、如果出问题，优先排查这几项

1. 前端是不是还连到旧地址 `8001`
   - 看 header 的绿色 tag
2. 后端 Swagger 是否能打开
   - `http://127.0.0.1:8081/docs`
3. Neo4j / Postgres 是否已启动
4. 前端是否已切到 Backend 模式
5. 浏览器 localStorage 里是否保留了旧的 API 地址

---

## 九、主干联调通过后，意味着什么

如果上面 3 条主干都通过，说明当前系统至少已经做到：

- 前端 backend mode 能稳定消费后端 Analysis 主链
- candidate / path / edge feedback 与 ledger 能闭环
- materials workspace 不再只是 build 后半段页面，而是能走真实 material library 主链

这就说明当前系统的前后端主干已经基本打通，可以继续往更完整的系统验收推进。
